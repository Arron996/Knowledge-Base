# 出票系统（Ticketing System）项目规则

## 一、项目概述

- **gds-ticketing-system**：火车出票系统（GDS 出票）。
- **三个相关项目**：
  - **admin fe**：admin 操作的前端，调用 admin be。主分支 `main`，测试分支 `base`。
  - **admin be**：admin 操作的后端，调用出票系统。主分支 `master`，测试分支 `base`。
  - **ticketing system**（本仓库）：出票系统。主分支 `master`，测试分支 `develop/base_v5_jdk21`。
- **命令行**：使用 **PowerShell** 语法。

---

## 二、项目结构（分层架构）

根包名：`com.ctrip.train.global.rail.gds.ticketing`。

| 模块 | 职责 | 主要目录/包 |
|------|------|-------------|
| **gds-ticketing-gateway** | 顶层入口：RPC 实现、HTTP 回调、MQ 消费、请求/响应映射 | `controller`、`rpc`、`mq`、`mapping`、`bean` |
| **gds-ticketing-business** | 业务逻辑：编排、策略、事件处理、Helper | `logic`、`strategy`、`processor`、`helper`、`event`、`service`、`cache` |
| **gds-ticketing-domain** | 领域模型与接口：POJO、常量、仓储接口、请求/响应 | `pojo`、`constant`、`repository`、`request`、`response`、`exception`、`metric` |
| **gds-ticketing-api** | 对外 API 定义：服务接口、请求/响应/常量/回调 | `pojo`、`request`、`response`、`constant`、`callback`、`common` |
| **gds-ticketing-infrastructure** | 基础设施：配置、仓储实现、代理、MyBatis、状态机、存储、工具 | `config`、`impl`、`repository`、`proxy`、`mybatis`、`statemachine`、`storage`、`utils` |
| **gds-ticketing-dal** | 数据访问层 | DAL 配置与实现 |

**依赖关系**：gateway → business → domain；business 依赖 api、infrastructure；infrastructure 依赖 domain、dal。API 与 domain 尽量被上层依赖，不依赖其他业务模块。

**各层常见子包**：
- **config**：QConfig/配置读取类。
- **constant(s)**：常量类。
- **metric**：监控与打点。
- **util(s)**：工具类。
- **beans**：Spring 配置（如使用）。
- 启动类：各模块根包下 `*Application.java`（如 `GdsTicketingSystemGatewayApplication`）。

## 三、开发规范

### 1. 代码复用与注释

- **优先复用**：编写代码时尽可能复用已有函数、流程，不要重复造轮子。
- **注释**：类、公开方法需添加注释，使用 **Javadoc** 格式。

### 2. 日志

- 在条件判断、分支、异常处理等**关键节点**插入日志。
- 使用 **@Slf4j** 或 **CLogger**，与项目现有用法保持一致。
- **方法入口**：方法开始处打 **log.info**，需包含**方法名**和**入参**（便于排查与追踪）。
- **方法出口**：方法正常结束处打日志，输出**结果**（如返回值摘要、成功标识等）；异常分支已在关键节点打日志即可，避免敏感信息泄露。

### 3. 隔离规范（README 约定）

- **简单业务**（如透传）：可以不做隔离，Request 层可使用 API 中的 Module，Response 给外部也可不做隔离。
- **复杂业务**：需要使用 **domain 层**进行隔离，避免把下层结构（如 VC 等）透传到业务内部；内部方法逻辑应使用 domain 层（领域模型、仓储接口等），不直接依赖 dal/api，防止业务结构被下层绑架。

### 4. 测试

- **单测/测试代码**：放在 **gateway 模块** 的 **test 目录** 下（`gds-ticketing-gateway/src/test`）。

### 5. 分支与功能开发

- 新功能需**单独建分支**，分支命名：`feature/yyyyMMdd_功能简述`（例如 `feature/20250128_adminOptimization`）。
- 新功能分支应从**主分支**创建，不要从开发/测试分支创建。
- 功能开发完成并自测通过后，按「三个项目 Git 提交流程」执行：提交 → push（首次可带 MR 创建）→ 签出测试分支并拉取 → 将功能分支 merge 到测试分支。

### 6. 内部逻辑方法的封装与分层依赖

- **内部逻辑封装**：业务内部逻辑方法的封装，**尽量不要直接依赖 DAL 层和 API 层**，而应优先使用项目的 **domain 层**（领域模型、仓储接口、领域服务等）。
- **原因**：domain 层承载领域概念与接口，便于复用、测试和分层清晰；直接依赖 dal/api 会使业务逻辑与数据访问、对外契约耦合，不利于维护与演进。
- **实践**：在 business、infrastructure 等模块中编写可复用的内部方法时，入参/出参与依赖尽量基于 domain 的 POJO、repository 接口；对 dal 的访问、对 api 的暴露应集中在边界（如 gateway 或明确的 facade），内部逻辑通过 domain 层编排。

### 7. 阿里巴巴开发手册（必须遵守）

- **总体要求**：开发规范遵循 **《阿里巴巴 Java 开发手册》**（及泰山版等后续版本）的约定，包括命名、格式、异常、并发、注释等。
- **修改影响其他流程时**：若本次修改**干涉到其他流程或已有逻辑**，应**优先新增**（新方法、新分支、新策略/处理器），而不是在原有流程上直接大改；必要时通过开关或策略路由区分新旧行为，避免牵一发而动全身、回归风险不可控。
- **示例**：新增一种退款场景时，可新增独立方法或策略类，在入口处按条件分发，而不是在现有退款主流程里堆叠大量 if-else 或直接改通用逻辑。

## 四、与 Cursor 协作时的注意点

- 修改或新增逻辑时，先在本规则与 `.cursor/rules/git-three-projects-workflow.mdc` 中确认模块归属与分支/提交流程。
- 涉及 Request/Response 与外部或下层系统对接时，遵守上述**隔离规范**，避免直接透传底层结构。
- 需要跑测试时，使用 **gateway** 模块下的 test 目录；命令行使用 **PowerShell**。
