说明：级别不是绝对的，真实考试以标准答案为准。这里给的是偏工程实践的参考判断。答题时最重要的是命中问题点，并说清影响和建议。

## 题目 1：线程池

参考级别：Major

问题点：

1. 直接使用 `Executors.newFixedThreadPool`。
2. 底层无界队列，任务堆积可能 OOM。
3. 线程池参数、线程名、拒绝策略、生命周期都不可控。

参考评论：

```
MAJOR: 这里直接使用 Executors.newFixedThreadPool 创建线程池，底层使用无界队列。导出任务处理慢或订单量较大时，任务会无限堆积，可能导致内存持续上涨甚至 OOM。
建议使用 ThreadPoolExecutor 显式配置核心线程数、最大线程数、有界队列、线程命名和拒绝策略，并将线程池统一交由配置类或公共线程池组件管理。
```
## 题目 2：异步等待

参考级别：Major

问题点：

1. `future.get()` 没有超时时间。
2. 下游慢或任务卡住时，请求线程无限阻塞。

参考评论：

```
MAJOR: 这里 Future.get 没有设置超时时间。下游查询慢、线程池堆积或任务异常未返回时，当前请求线程会一直阻塞，可能导致请求线程耗尽并引发级联故障。
建议使用 future.get(timeout, TimeUnit) 并处理 TimeoutException，同时对超时任务执行取消、降级或失败返回。
```
## 题目 3：多个条件判断

参考级别：Medium

问题点：

1. 多个状态值用 `||` 串联，可读性差。
2. 后续新增状态容易漏改。

参考评论：

```
MEDIUM: 这里多个状态条件通过 || 串联，可读性和可扩展性较差，后续新增可退款状态时容易漏改。
建议使用 Set、枚举字段或 isRefundable 方法表达业务语义，避免魔法值散落在业务代码中。
```
补充：如果这个判断直接影响退款正确性，并且状态缺失会导致错误退款或拒退，严重性可能升高。

## 题目 4：敏感日志

参考级别：Major

问题点：

1. 完整打印支付请求。
2. 包含手机号、证件号、银行卡、token。
3. 日志平台可见范围广，存在敏感信息泄露风险。

参考评论：

```
MAJOR: 这里直接打印完整 paymentRequest，包含手机号、证件号、银行卡信息和支付 token 等敏感字段，日志平台可见范围较广，存在敏感信息泄露风险。
建议只打印 orderId、requestId 等必要定位字段，对手机号、证件号、token 等敏感字段脱敏或禁止输出。
```

## 题目 6：MQ 消费

参考级别：Major

问题点：

1. 退款消费缺少幂等。
2. MQ 可能重复投递。
3. 先调用退款再插入记录，失败时状态不一致。

参考评论：

```
MAJOR: 该退款 MQ 消费逻辑缺少幂等控制。消息重复投递、消费超时重试或服务重启后可能重复调用退款接口，造成重复退款等资金风险。
建议基于 refundId/orderId 等业务唯一键建立幂等记录，执行退款前判断处理状态，并保证外部退款调用和本地状态更新的一致性。
```

## 题目 8：事务中调用远程服务

参考级别：Major

问题点：

1. 数据库事务中调用供应商。
2. 下游慢会长时间占用数据库连接。
3. 远程成功、本地回滚会造成跨系统不一致。

参考评论：

```
MAJOR: 这里在数据库事务中执行远程调用，事务边界过大。供应商接口慢或超时时会长时间占用数据库连接；远程调用成功但本地事务回滚时，还会造成本地订单和供应商状态不一致。
建议缩小本地事务范围，通过状态机、本地消息表或补偿任务编排远程调用，实现最终一致性。
```

## 题目 12：权限校验

参考级别：Major

1. 用户端接口只按 `orderId` 查询。
2. 没校验当前用户是否拥有该订单。
3. 存在越权访问风险。

参考评论：

```

建议在查询条件中加入 userId/tenant/channel 等权限范围校验，对无权限访问返回明确错误。
```

2. 仍然更新本地状态为取消。
3. 本地和下游状态不一致。

参考评论：


## 题目 14：命名

参考级别：Medium

问题点：

1. `type`、`f`、`map` 语义弱。
2. `"1"`、`"Y"` 魔法值。
3. 如果这是核心入口，后续维护风险较高。

参考评论：

```
MEDIUM: 该方法参数命名过于泛化，type/f/map 无法表达具体业务语义，同时直接使用 "1"、"Y" 等魔法值判断，阅读和维护成本较高。
建议使用有业务含义的参数对象，并将状态值抽取为枚举或常量。
```

## 题目 17：CompletableFuture 默认线程池

参考级别：Major

问题点：

1. `supplyAsync` 未指定业务线程池，默认使用 commonPool。
2. `join()` 没有超时。
3. 下游慢可能阻塞并影响其他共用 commonPool 的任务。

参考评论：

```
MAJOR: 这里 CompletableFuture.supplyAsync 未指定业务线程池，会使用 ForkJoinPool commonPool，容易与其他异步任务相互影响；同时 join 没有超时，下游慢响应时当前线程可能长期阻塞。
建议指定受控的业务线程池，并使用 orTimeout/completeOnTimeout 等方式设置超时和失败处理。
```

## 题目 19：分布式锁

参考级别：Major

问题点：

1. 锁释放没有放在 finally。
2. `processRefund` 抛异常会导致锁不释放。
3. 退款流程被长期阻塞。

参考评论：

```
MAJOR: 这里加锁后没有在 finally 中释放锁，processRefund 抛异常时锁无法释放，可能导致该订单后续退款流程长期阻塞。
建议加锁成功后使用 try-finally 保证释放，并设置合理锁超时时间和 owner token，避免误释放其他线程持有的锁。
```
## 题目 20：大方法

参考级别：Major

问题点：

1. 参数过多，多个 `String` 和 `Map` 弱语义。
2. retry/delete/operationType 多套分支混在一个方法。
3. 核心动作通过 if-else/switch 堆叠，扩展风险高。

参考评论：

```
MAJOR: 该 handle 方法入参过多且分支复杂，retry、delete、CREATE/UPDATE/REFUND 等多个业务动作混在同一个方法中，职责不清晰，可读性和可维护性较差。
建议封装业务请求对象表达参数语义，并通过策略模式或独立处理器按操作类型路由，避免继续堆叠 if-else/switch。
```
## 总结

容易判错的点：

1. 格式差通常是 Minor，但复杂 lambda 里如果有资源泄漏、异常吞掉，就要另提 Major。
2. 条件复杂通常是 Medium，但如果控制支付、退款、权限、状态机，可能升为 Major。
3. 日志组件不统一通常是 Medium，但敏感日志是 Major。
4. 命名差通常是 Minor/Medium，但如果伴随参数过多、Map 透传、大量分支，可能是设计类 Major。
5. 线程池、无超时、资源泄漏、幂等缺失、越权、事务远程调用，优先按 Major 排查。
