## 二、代表交付（按模块）

### VCC 实付对账

- TripLink 授权结果回调 → 卡号反查 `order_vcc_info` → 抛 Q 给订单算负利润
- 退款无实时回调 → job 捞「退款中」+ T+1 清算报表补数
- `needVcc` / `isVCOrder` 配置切换边界处理
- 详见 [[领域/VCC实付对账模型]]

### applyRefund 迁移

- Request → InnerRequest → SupplierApplyRefundRequest 契约下沉
- 子单异步调 VC，聚合 refundable / partialRefundable
- 单票 / 多票 / 整单退；uktis 去程返程维度
- 详见 [[领域/退票可退维度问题集]]

### 通票 / X 产品人工运维

- 状态机：order `100→200→400→600`；subXorder `200→400→401→600`
- ManualTicketResult、ConfirmXProductRefund、ManualRefundResult
- 主退 / 子退表设计（`supplier_master_xorder_refund` 等）
- 详见 [[领域/通票X产品人工出退改]]

### 老票台迁移接口

- **createInvoice**：renfe 同步 urlList；iryos 供应抛 Q 发邮件；退改置发票失败
- **obbat obtainTicket**：obtain 后才可退；update → job → queryVc → upload → send
- **changeTicketOption**：uktis ETICKET ↔ TOD 票种转换

### Offline 全栈

- FormData 上传 PDF + JSON 序列化集合参数
- 列表搜索、出票弹窗、日铁多文件上传
- 详见 [[工程/Offline前后端联调套路]]

## 三、领导力

### 思考力

解决 VCC 时先问需求背景，独立思考后再执行；对金额类异常数据敏感，主动定位不合理点；受阻时积极找相关方换方案（如退款无实时回调改走 T+1 报表）。

### 执行力

VCC 链路日志埋点 + 定时监控未处理问题；主动跟代码全链路；多条迁移接口按期交付并加重试兜底。

### 自驱力

出票列表前端充分考虑客服体验，按反馈迭代交互；了解新票台规划，主动提出改进与工具方案。
